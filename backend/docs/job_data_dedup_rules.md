# 岗位数据去重规则说明

岗位数据导入后，清洗阶段按以下规则生成去重主键 `canonical_key`，并按批次记录清洗结果，支持后续回溯。

## 去重规则优先级

1. 若岗位存在源系统提供的 `职位编码 / 岗位编码`，使用：
   - `公司标准名 + 岗位标准名 + 职位编码`
2. 若岗位编码缺失，自动回退为：
   - `公司标准名 + 岗位标准名 + 工作城市 + 月薪下限 + 月薪上限`

上述组合会进一步做文本规范化，并生成 SHA1 摘要作为稳定的 `canonical_key`。

## 标准化细则

- 公司名：去除多余空格、括号补充说明等噪声内容。
- 岗位名：去除“岗位 / 职位”等噪声词，统一空白与常见格式差异。
- 城市：从工作地址中提取标准城市，提取失败时写入 `未知城市`。
- 薪资：统一折算为月薪区间，支持 `K`、`万/月`、`万/年`、`元/天`、`元/小时` 等常见格式。

## 去重行为

- 同一批次内，遇到相同 `canonical_key` 的记录，保留首条进入标准表。
- 与历史批次相比，若 `canonical_key` 已存在，也会标记为重复并跳过，不再进入标准表。
- 重复记录会保留在 `job_postings_raw`，并将 `clean_status` 标记为 `duplicate`。
- 每条重复识别结果都会写入 `job_cleaning_logs`，记录首条来源 `raw_id` 与对应 `canonical_key`。

## 可解释性与可复现性

- 去重只依赖标准化后的岗位字段，不依赖随机值。
- 相同原始岗位内容在不同机器、不同时间重复清洗，得到的 `canonical_key` 一致。
- 批次号 `batch_id` 用于区分不同导入批次，方便清洗日志和结果回溯。

## 典型适用场景

- 官方岗位表中存在重复行
- 多渠道导入后同一岗位重复出现
- 源数据缺失职位编码时，仍能通过岗位关键信息识别重复
